# BirdWeather HomeOps

BirdWeather ingestion and Home Assistant display pipeline for the PUC station.

## Current Flow

- BirdWeather PUC uploads detections, soundscapes, and sensor readings to BirdWeather.
- `birdweather-ingester` runs as a long-lived Kubernetes `Deployment` in the `home` namespace.
- The worker polls BirdWeather every 5 minutes.
- Detection/species/sensor/weather data is stored in the shared Crunchy Postgres cluster.
- Unique soundscape audio clips are downloaded once per `soundscape_id` and stored in Postgres.
- The worker publishes Home Assistant MQTT discovery/state messages for dashboard entities.
- The Weaver Home Assistant dashboard consumes those MQTT entities.

## Kubernetes Pieces

- Flux Kustomization: `kubernetes/apps/home/birdweather/ks.yaml`
- App manifests: `kubernetes/apps/home/birdweather/app/`
- Container source: `apps/birdweather-ingester/`
- Image: `ghcr.io/boundcorp/birdweather-ingester:latest`
- Database user/database: `birdweather`, declared in `kubernetes/apps/database/crunchy/cluster.yaml`
- Media PVC: `birdweather-media`, mounted at `/media/birdweather` in Home Assistant and the ingester.

The GHCR package is private right now. The live cluster uses an image pull secret named `ghcr-boundcorp-pull` in the `home` namespace.

## Secrets

`secret.sops.yaml` contains:

- `token`: BirdWeather station token
- `station-id`: BirdWeather station id

The database connection secret is generated by Crunchy as `postgres-pguser-birdweather` in `database`. A copy must exist in `home` for the worker.

Manual copy pattern:

```bash
KUBECONFIG=./kubeconfig kubectl get secret -n database postgres-pguser-birdweather -o json \
  | jq 'del(.metadata.ownerReferences, .metadata.resourceVersion, .metadata.uid, .metadata.creationTimestamp, .metadata.managedFields) | .metadata.namespace="home"' \
  | KUBECONFIG=./kubeconfig kubectl apply -f -
```

## Database Schema

The worker creates and owns schema `birdweather`.

Main tables:

- `species`: species metadata from BirdWeather.
- `detections`: one row per BirdWeather detection id.
- `sensor_readings`: PUC environmental/light/system/location snapshots plus cloud weather data.
- `station_snapshots`: raw station snapshot payloads.
- `soundscape_assets`: one audio blob per unique BirdWeather `soundscape_id`.
- `poll_runs`: per-poll status and error history.
- `ingest_state`: cursor/state data for incremental polling.
- `species_facts`: curated or generated short facts for each species.
- `species_photos`: sourced reference photos with attribution/license metadata.
- `species_artifacts`: generated or derived images, including future bird drawings.
- `daily_species_cards`: composed card images for species seen on a given local date.
- `species_hourly_stats`: per-species detection counts by local date/hour.
- `species_monthly_stats`: per-species historical detection counts by month/hour.

Species enrichment:

- The worker enriches species missing facts/photos after each poll.
- Current source is Wikipedia/Wikimedia Commons, which requires no API key.
- Summary facts are sourced from Wikipedia page summaries.
- Photos are sourced from Wikimedia page image metadata where attribution/license metadata is available.
- Enrichment is controlled by `ENRICH_SPECIES`, `MAX_SPECIES_ENRICHMENTS_PER_POLL`, and `MAX_PHOTOS_PER_SPECIES`.

Media export:

- Source photos are exported under `/media/birdweather/source-photos/<species_id>/`.
- Composed cards for species seen today are exported under `/media/birdweather/bird-cards/`.
- The card export folder is pruned after each generation pass, so birds not seen today are removed from the Home Assistant media source.
- The same files are visible in Home Assistant's media browser under the `birdweather` folder.
- The PVC uses `nfs-titan-nvme`, so the backing files are also inspectable on Titan's NFS storage.
- Postgres remains the source of truth for metadata and also stores the composed PNG bytes.

Home Assistant must register the mounted media path before WallPanel can browse it:

```yaml
homeassistant:
  media_dirs:
    birdweather: /media/birdweather
```

WallPanel should point at the registered media source, not the container filesystem path:

```yaml
wallpanel:
  image_url: /birdweather/bird-cards
```

Audio storage details:

- Audio is stored as `bytea` in `soundscape_assets.audio`.
- Detections keep `soundscape_start_time` and `soundscape_end_time`.
- Multiple detections can reference the same full FLAC asset.
- `MAX_AUDIO_DOWNLOADS_PER_POLL=20` avoids long startup/backfill polls.
- `MAX_AUDIO_BYTES=26214400` bounds each individual asset.

## Home Assistant Entities

MQTT discovery currently creates:

- `sensor.birdweather_detections_today`
- `sensor.birdweather_species_today`
- `sensor.birdweather_latest_species`
- `sensor.birdweather_latest_detection_at`

The Weaver dashboard has a BirdWeather card that uses these entities and renders the `species` attribute from `sensor.birdweather_detections_today`.

## Roadmap

### 1. Bird Facts

Goal: enrich each detected species with a reusable library of short facts.

Initial implementation:

- `species_facts` exists and is populated from Wikipedia summaries.
- Next pass should add more structured fact providers to reach several dozen concise facts per species.
- Prefer stable fact categories such as behavior, call/song, migration, nesting, diet, habitat, range, field marks, conservation, and local seasonal notes.

Possible schema:

- `species_id`
- `fact`
- `category`
- `source_name`
- `source_url`
- `generated_by`
- `reviewed_at`
- `created_at`

### 2. Species Photos

Goal: source real reference photos for each species.

Initial implementation:

- `species_photos` exists and is populated from Wikimedia image metadata.
- Store photo URL, thumbnail URL, photographer/attribution/license, source, dimensions, and preferred display rank.
- Do not hotlink blindly in generated cards if the source license requires attribution or local caching.
- Use photos as references and fallback display assets, not as generated-art replacements.

### 3. Home Assistant Media Library

Goal: generate and expose beautiful daily Bird Cards for Browser Mod screensavers.

Proposed flow:

1. On first detection of a new species, generate several reusable bird illustrations.
2. Store generated assets once per species/style to control cost.
3. Styles to start with:
   - watercolor field-guide plate
   - oil portrait
   - graphite sketch
   - ink-and-wash
4. Store generated assets in Postgres metadata plus object/file storage for the image bytes.
5. On each new detection, regenerate the daily Bird Card for that species.
6. Publish today's cards into Home Assistant's media-accessible path.
7. Point Browser Mod screensaver media at that HA media library folder.

AI provider note:

- Do not commit or reuse existing personal API keys for image generation.
- When this feature is implemented, choose the image model/provider at that time based on current quality, cost, licensing, and API ergonomics.
- Ask for a fresh dedicated API key before enabling generation.
- Store that key in a new SOPS secret and keep generated-art calls behind an explicit feature flag.

Bird Card content:

- Random generated drawing for the species.
- Common and scientific name.
- A rotating selection of short facts.
- Detection count today.
- Hours sighted today.
- Hours commonly sighted this month.
- Months commonly seen historically.
- Latest detection time.

Compositor approach:

- Python image compositor using Pillow.
- Fixed output aspect ratios for screensaver surfaces.
- Keep templates deterministic and readable before adding many styles.
- Generate cards into `daily_species_cards` and atomically replace the HA media files.

Suggested future tables:

- The metadata/card/stats tables now exist as scaffolding.
- Next work is populating `species_facts` and `species_photos`.
- After that, implement generated `species_artifacts` and `daily_species_cards`.

## Design Notes

- Keep Home Assistant as a display consumer, not the source of truth.
- Keep raw detections/audio/sensor data in Postgres.
- Publish only dashboard-ready summaries to HA over MQTT.
- Generate expensive AI assets once per species/style.
- Regenerate cheap composed cards as data changes.
